# نظام قياس رضا المستفيدين (Library Satisfaction System - LSS)

نظام ويب مستقل يعمل على جهاز واحد عند بوابة خروج المكتبة الجامعية، لقياس رضا المستفيدين عن الخدمات بطريقة بسيطة وسريعة، مع لوحة تحكم إدارية كاملة للإحصاءات والتقارير والإعدادات.

النظام مستقل تماماً عن أي نظام مكتبات آخر، ولا يرتبط ببيانات شخصية للمستفيدين.

## ⚠️ انحرافان تقنيان متعمدان عن وثيقة المتطلبات (باختصار)

قبل أي شيء، إليك التعديلين اللذين أجريتهما على قائمة التقنيات المطلوبة، وسبب كل منهما:

### 1. قاعدة البيانات: `sql.js` بدل `sqlite3`

حزمة `sqlite3` (وكذلك البديل الشائع `better-sqlite3`) حزم native تحتاج تصريف ثنائي (binary) مطابق تماماً لنظام تشغيلك ونسخة Node لديك. إن لم يوجد ملف جاهز مطابق، تحاول npm تصريفه من المصدر، وهذا يتطلب تحميل ملفات رأس Node من `nodejs.org` — **وهو بالضبط نوع العطل الذي واجهته سابقاً مع محرك Prisma** (خطأ DNS لـ `binaries.prisma.sh`). اختبرت كلا الحزمتين فعلياً وفشلتا في بيئتي بنفس النمط من العطل.

`sql.js` هو SQLite مُصرَّف إلى WebAssembly: من منظور npm هو JavaScript خالص، لا يوجد فيه أي شيء يحتاج تصريفاً أو تحميلاً إضافياً بعد `npm install`. الفارق الوحيد: يبقي قاعدة البيانات في الذاكرة، ويجب حفظها صراحة إلى القرص بعد كل عملية كتابة — وهذا ما يقوم به `persist()` في `database/db.js` تلقائياً بعد كل عملية تعديل، فلا تفقد أي بيانات عند إعادة تشغيل الخادم.

### 2. تصدير PDF: طباعة المتصفح بدل PDFKit

اختبرت `PDFKit` فعلياً مع خط Cairo عربي — النتيجة: الحروف العربية تُرسم منفصلة وغير متصلة بالشكل الصحيح (PDFKit لا يقوم تلقائياً بعمليات "تشكيل" الحروف العربية أو إعادة ترتيبها من اليمين لليسار). بدل إضافة مكتبات تشكيل نصوص عربية إضافية معقدة، استخدمت طباعة المتصفح (`window.print()` في `report.html`) والتي تعرض العربية بشكل صحيح فوراً دون أي تعقيد. `PDFKit` غير مثبَّت في المشروع لعدم استخدامه فعلياً؛ إن أردت اعتماده لاحقاً لأي غرض آخر (تقارير بالإنجليزية مثلاً)، أخبرني.

كل بقية التقنيات (`Express.js`, `bcrypt`, `express-session`, `Chart.js`, `xlsx`) مطابقة تماماً لما طلبته، واختبرتها جميعاً فعلياً وتعمل بنجاح.

### ملاحظة أمان بخصوص `xlsx`

نسخة `xlsx` المتاحة مجاناً على npm بها ثغرتان معروفتان غير مصححتين (Prototype Pollution وReDoS) تظهران عند تحليل ملفات Excel **واردة من مصدر غير موثوق**. بما أن استخدامنا هنا هو **التصدير فقط** (كتابة بيانات نعرفها نحن، وليس قراءة ملفات يرفعها الغير)، فمسار الثغرة غير مستخدَم عملياً في هذا التطبيق — لكن أذكرها هنا للشفافية الكاملة.

## المتطلبات

- Node.js **22.5 أو أحدث** (يُستخدم للتحقق من عمل بعض الأدوات الداخلية فقط، لا علاقة له بقاعدة البيانات؛ تحقق من إصدارك عبر `node -v`)
- npm

## التشغيل

```bash
npm install
npm start
```

ثم افتح المتصفح على: `http://localhost:3000`

**بيانات دخول الإدارة الافتراضية:**
- اسم المستخدم: `admin`
- كلمة المرور: `admin@2026`

غيّرها من صفحة الإعدادات بعد أول تسجيل دخول.

### وضع الكشك (Kiosk) الفعلي على جهاز العرض

طلب فتح شاشة التقييم بملء الشاشة تلقائياً موجود في `survey.js` (يحاول ذلك عند تحميل الصفحة)، لكن المتصفحات لا تسمح عادة بملء الشاشة برمجياً إلا مباشرة بعد نقرة المستخدم على نفس الصفحة، فقد لا يعمل هذا تلقائياً في كل المتصفحات. **الطريقة المضمونة** المتبعة عملياً في أكشاك المكتبات هي تشغيل المتصفح نفسه في وضع kiosk من سطر الأوامر، مثلاً:

```bash
# Google Chrome / Edge
chrome --kiosk http://localhost:3000
```

### إضافتان صغيرتان على الهيكل المطلوب

وثيقتك لم تُدرج مجلد `models/` ولا صفحة/تحكم منفصل لإدارة الأسئلة صراحة، لكن:
- **`models/`**: مطلوب فعلياً لتحقيق "MVC" الذي طلبته في متطلبات البرمجة (قسم 20) — بدونه تتحدث الـ Controllers مباشرة مع SQL، وهذا ليس MVC حقيقياً.
- **`controllers/questionController.js` + `routes/questions.js`**: قسم "إدارة الأسئلة" (رقم 15) في وثيقتك يحتاج مكاناً يعيش فيه؛ فصلته عن `settingController` لتجنب خلط الاهتمامات (Separation of Concerns)، وواجهته مدمجة داخل `settings.html` نفسها كقسم "إدارة الأسئلة".

## مقياس التقييم قابل للتعديل

الدرجات لم تعد ثابتة عند (جيد=3، مقبول=2، ضعيف=1). من صفحة الإعدادات → قسم "مقياس التقييم"، يمكن للمسؤول:
- تغيير عدد المستويات (مثلاً 5 مستويات: ممتاز-جيد-متوسط-مقبول-ضعيف).
- تغيير نص أي مستوى ولونه ورمزه التعبيري (مثلاً لمقياس ليكرت: موافق بشدة-موافق-محايد-غير موافق-غير موافق بشدة).
- إعادة ترتيب المستويات وحذف/إضافة مستويات (بحد أدنى مستويين).

كل الحسابات (متوسط كل سؤال، تصنيف كل زيارة، شاشة النتيجة، مخطط التوزيع، حالة كل سؤال في التقرير) مبنية على "أقرب مستوى مُهيَّأ" للمتوسط الفعلي، وليست عتبات ثابتة مبرمجة لثلاث درجات فقط — فتعمل بشكل صحيح مهما كان عدد المستويات أو نصوصها.

## قاعدة البيانات

الجداول الأربعة التي حددتها (`users`, `questions`, `statistics`, `daily_summary`) موجودة بنفس الأسماء والحقول تقريباً، مع تعديل بسيط على `daily_summary` (انظر أدناه). أضفت:
- **`meta`** (key/value): اسم المكتبة، مسار الشعار، مدة عرض شاشة النتيجة، متوسط زمن الإجابة، تاريخ آخر تقييم.
- **`rating_levels`**: مقياس التقييم القابل للتعديل (النص، القيمة الرقمية، اللون، الرمز التعبيري، الترتيب).
- **`daily_level_counts`**: عدد الزيارات لكل مستوى تقييم في كل يوم — حلّت محل أعمدة `good/normal/bad` الثابتة في `daily_summary` الأصلي، لأن عدد الأعمدة الثابتة لا يتسع لعدد متغيّر من المستويات.

## كيف تم اختباره

كل هذا اختُبر فعلياً (وليس افتراضاً) في بيئة التطوير، على قاعدة بيانات SQLite حقيقية:

- تسجيل الدخول الصحيح والخاطئ، وحماية كل الصفحات والمسارات الإدارية (تُعيد 302/401 دون جلسة).
- تعبئة استبانة كاملة (12 سؤالاً) والتحقق الرياضي من: المتوسط المحسوب، التصنيف الصحيح (😊/😐/😞)، تحديث الإحصاءات لكل سؤال، تحديث ملخص اليوم، متوسط زمن الإجابة.
- لوحة التحكم: صحة أرقام المشاركين (اليوم/الأسبوع/الشهر)، أفضل/أضعف خدمة، تحميل الرسوم البيانية الثلاثة ببيانات حقيقية.
- التقارير: تصدير CSV (بترميز UTF-8 BOM يعرض العربية بشكل صحيح في Excel) وتصدير Excel فعلي (ملف .xlsx صالح تم التحقق منه).
- إدارة الأسئلة: إضافة، تعديل، حذف، تفعيل/تعطيل، تغيير الترتيب — كل عملية اختُبرت عبر الواجهة الفعلية.
- الإعدادات: تغيير اسم المكتبة، رفع شعار (PNG حقيقي)، تغيير كلمة المرور، تعديل مدة شاشة النتيجة.
- **النسخ الاحتياطي والاستعادة بدورة كاملة**: تم إنشاء نسخة احتياطية، إعادة ضبط كل الإحصاءات (تصفير)، ثم استعادة النسخة الاحتياطية والتحقق من عودة البيانات بالضبط كما كانت قبل التصفير.
- تشغيل الصفحات الست فعلياً داخل متصفح محاكى (jsdom) والتحقق من عدم وجود أي خطأ برمجي JavaScript في أي منها.
